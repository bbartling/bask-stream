Install And Connect

  1. Install the compiled baskStream module jar on the station host and restart Niagara.
  2. Add the BASkStreamService to the station, then verify GET /stream/health responds after login.
  3. Create or choose a Niagara user with only the station permissions your external app needs.
  4. From the app, authenticate to the station, open wss://<station>/stream, then call capabilities.
  5. Use browse or search for discovery, read for snapshots, and subscriptions for live screens.

External Graphics Pattern

Keep station structure and metadata cached in your app. Request metadata during initial discovery and refresh passes, not on every live-value update.

  1. Browse shallow roots such as slot:/Drivers, slot:/Schedules, and history ORDs as needed.
  2. Store node identity with ord, slotPath, parent, type, features, operations, and metadata evidence.
  3. When a graphic opens, call replace_subscriptions for the points on that screen.
  4. When the graphic closes, call release_subscriptions. Leases clean up missed releases.
  5. Keep alarms subscribed separately when the app needs global alarm awareness.

Reverse proxiesPreserve cookies, TLS, and Upgrade: websocket headers if routing through a gateway.

Scale Rules

  1. Use shallow browse calls and drill in; avoid deep all-station metadata pulls on every startup.
  2. Use metadata: "full" for discovery or refresh and omit metadata for routine tree navigation.
  3. Use grouped subscriptions for active views so the server can diff and share point listeners.
  4. Read histories and schedules on demand unless a screen actually needs them.
  5. Use subscription_status during commissioning to confirm point counts, groups, and leases.

Using This Companion App

The Test Console tab connects to a real station, browses from the typed Root ORD, shows metadata when available, reads selected point values, and subscribes to the selected point with grouped subscriptions when the module supports them.

The local helper is only for this standalone file. Production apps should implement their own Niagara login flow, trust model, and MessagePack WebSocket client.

The full protocol reference lives in docs/THIRD_PARTY_API.md.
